# Security residuals (internal)

Known security findings that are **not yet closed in code**, why, and when to recheck. Internal — NOT
in the public mkdocs nav. Process: a residual is either (a) deferred-by-design to a scoped follow-up,
or (b) unfixable at Verel's layer (upstream dependency / OS) — in which case, if upstream is open
source, we open an issue + PR and shepherd it to merge. Recheck on the date and close when fixed.

| ID | Finding | Severity | Why open / mitigation | Recheck |
|----|---------|----------|-----------------------|---------|
| R-002 | **GateRun pod DNS egress is unrestricted** (`src/verel/operator/jobs.py` netpol: port 53 to any destination). Untrusted repo code could DNS-tunnel/exfiltrate through the cluster resolver. | Low | Inherent to plain NetworkPolicy (no egress-DNS firewall). **Bounded:** nothing of value (cloud metadata / API server) listens on :53, so it is exfil-only — no pivot. `:443` egress to the public internet is already permitted by necessity (the git clone), so DNS adds no capability an attacker doesn't already have. Closing needs an egress-DNS policy engine (Cilium FQDN policy / a DNS proxy) which is a cluster-level control, not the operator's. | When the chart documents a Cilium/FQDN-policy option. |
| R-003 | **No per-pod PID cap on the GateRun Job** (Kubernetes has no pod-spec PID field). A fork bomb in untrusted code is bounded only by the memory cgroup + node `--pod-max-pids`. | Low | **Bounded:** the 2Gi memory cgroup OOM-kills before PID exhaustion, and untrusted repo code additionally runs under the in-container bwrap `--unshare-all` PID namespace. Documented as a cluster prerequisite (set kubelet `--pod-max-pids`) in the `jobs.py` netpol docstring + the k8s install docs. | If/when a Pod-level PID limit lands in core k8s. |
| R-005 | **Terraform actuator error text may echo secret values** (`src/verel/actuators/terraform.py`: `ActResult.reason` / report summaries carry `err[:200]` from the terraform CLI, which can contain sensitive variable values in some error messages). | Low | **Bounded:** truncated to 200 chars and never written to the brain/logs by the actuator itself; the caller decides where reasons surface. Closing fully needs structured-error redaction (strip `value=...` spans) — a Phase-5 hardening item. | Phase 5 (effective-access + redaction pass). |
| R-006 | **Capture-C IAM tool interception is a denylist** (`iam_action_class`): an IAM-widening tool name not in `_IAM_MUTATING` won't auto-escalate to IRREVERSIBLE. | Low | **Bounded:** the gateway already fail-closes an *unknown-shape* tool to CONSEQUENTIAL (verdict-gated, not auto-forwarded), and the IaC-plan IAM path (Capture A) catches IAM changes deterministically regardless of tool name. The denylist is a convenience escalation on top, not the only control. | As new cloud SDK verbs surface; consider an allowlist-per-host model. |
| R-007 | **Actuator trusts `repo`/cwd** (the working directory is passed as the subprocess cwd). A malicious cwd is an operator-supplied input, not attacker-influenced. Also covers the narrow `plan()` window where the escalation class is read from `show -json` while the digest binds the planfile bytes — a mid-`plan()` swap of a file in a trusted cwd. | Info | **By design:** the operator chooses which working dir to plan/apply; argv values (binary, planfile, and now ALL grader paths/values) ARE charset-validated against option/shell injection. | n/a (operator-trust boundary). |
| R-008 | **Cloud-CLI subprocesses inherit the full parent `os.environ`** (`access_verify.subprocess_env_runner`) and tools are resolved via `PATH` (not absolute). A parent process whose env is attacker-influenced could redirect `aws`/`gcloud`/`az`/`terraform` (PATH hijack) or change behavior via `AWS_CONFIG_FILE`/`CLOUDSDK_*`. | Low | **Bounded:** requires control of the parent process environment (already a strong position). The verify path is opt-in/online and operator-invoked. Closing fully needs a minimal allowlisted child env + absolute tool paths — a follow-up hardening. | When the verify path is wired into an unattended/CI runner. |
| R-010 | **Plan-binding does not stop a same-uid adversary who controls the working dir** — for EITHER the apply-time re-open OR the plan-time classification. `act()` digest-checks then `terraform apply` re-opens by path (sub-ms race); and within `plan()` the digest, `terraform show` (classification/grade), and the receipt read are separate path reads, so a writer racing `show` can make a destructive/IAM-widening plan classify CONSEQUENTIAL (skipping human approval) while the evil digest is bound. | Low | **Mitigated, then scoped:** `plan()` re-digests before/after `show` and refuses on a mid-plan digest change; `act()` re-checks the digest immediately before apply. The irreducible residual is that `terraform show`/`apply` re-open the file by path under a *same-uid* attacker — no in-process check can bind it; closing it needs privilege separation (actuator in a different uid / container = the `actel` OS-isolation story). The code comments state this scope honestly rather than claiming the in-process guard closes it. Consistent with R-007 (cwd is operator-trusted). | When the actuator runs untrusted-producer plans without OS isolation. |
| R-012 | **IAM-risk rule coverage is a pattern-based deny-list with a long tail.** The detection ENGINE (policy-doc parsing incl. NotAction / mid-string wildcards / structural & generic extraction / `kind: List` unwrap, privesc-glob match, public-principal, RBAC) survived a 5-round red-team with no residual *engine* bypass. But each cloud has hundreds of resource types and privesc primitives; coverage is enumerated in sets (`_PRIVESC_ACTIONS`, `_ADMIN_*`, `_PRIVESC_*`, `_IAM_TYPE_SUBSTRINGS`, `_DANGEROUS_RBAC_RESOURCES`). Known still-missing coverage (add as needed): AWS PassRole-to-compute pairs (`lambda:CreateFunction`+iam:PassRole, `ec2:RunInstances`, `cloudformation:CreateStack`) are *flagged* (PassRole gates ERROR) but not elevated to CRITICAL for the compound; more Azure built-in privesc roles. **Closed in round 9:** non-AWS open-ingress (`google_compute_firewall`, `azurerm_network_security_rule` → `OPEN_INGRESS`), non-policy public exposure (`aws_s3_bucket_acl` public-read → `PUBLIC_ACL`; RDS/Redshift `publicly_accessible` → `PUBLIC_DB_ENDPOINT`). | Med | **By design** for a static gate: it is best-effort and ever-extending, not a complete model of every cloud grant. New types are added to the sets over time; `trivy`/`checkov`/`parliament`/`cloudsplaining` (also wired) catch overlapping classes. Fails *open* on an unmodeled type (no issue), so pair with the cloud scanners for defence-in-depth. | Ongoing — extend the sets as new privesc patterns surface. |
| R-013 | **IAM `Condition` blocks are not evaluated.** A resource policy with `Principal:"*"` plus a scoping `Condition` (`aws:PrincipalOrgID`, `aws:SourceArn`) is still flagged `PUBLIC_PRINCIPAL`. | Info | **Conservative-fail (over-flag), not a miss** — safe for a gate; adds review noise on legitimately-scoped public-principal policies. A future enhancement can treat recognized scoping conditions as mitigating. | When false-positive noise on scoped policies becomes a problem. |
| R-009 | **Cred files (`~/.config/AWS/rootkey.csv`, `~/.config/gcp/*.json`) tolerated at group/world-readable mode.** | Low | **Mitigated:** symlink + foreign-owner are hard-failed (`available=False`) — the substitution/swap attacks; loose *mode* (others can read the user's own file) is surfaced as a `verel doctor` warning + `verify-access` stderr advisory rather than hard-failed, so a pre-existing `0644` file keeps working. Operator can `chmod 600`. | Recheck if cred files move into an automated/multi-tenant context. |
| R-014 | **Staged-apply / unknown `count`·`for_each` can defer a resource out of one plan.** A resource whose `count`/`for_each` is unknown-at-plan, or a planfile built with `-target`, omits resources from `resource_changes` entirely — each *individual* plan looks clean while a later apply adds the grant. (Round-6 finding P3.) | Low | **Mitigated, then scoped:** modern terraform *errors* at plan on an unknown `count`/`for_each` ⇒ `plan()` fails closed to IRREVERSIBLE; the computed-content path (`UNKNOWN_IAM_CONTENT`) now also gates a deferred IAM field that *does* appear with `after_unknown`. The irreducible residual is the multi-apply sequence where each plan is individually clean and complete — no single-plan grader can see across applies. A config-vs-changes cross-check (flag a configured resource absent from `resource_changes`) is a candidate follow-up but risks false positives on legitimately-targeted plans. | When a plan-set / multi-apply diffing story is added. |
| R-015 | **`watch()` can read false-convergence** (`src/verel/actuators/terraform.py`). A resource with `lifecycle { ignore_changes = [policy] }`, or a provider that under-reports drift, shows no diff on the post-apply re-plan even when live state diverged — so `watch()` returns PASS "converged". (Round-6 P4.) | Low | **Bounded:** exit-code handling itself is fail-closed (only literal rc==0 reads as converged; rc 1/timeout/exec-error → errored FAIL). The gap is terraform's own drift visibility, which `ignore_changes` deliberately suppresses — inherent to plan-based convergence, not something an out-of-band check at this layer can recover. Pair with the effective-access verifier (reads the cloud's *actual* grant, not the plan) for IAM-relevant drift. | When act-then-verify gains a live-state (not re-plan) convergence check. |
| R-016 | **`data.external` / external-program data sources run at plan/refresh, invisible to the grader.** A `data "external"` block executes an arbitrary program every refresh with side effects, and data-source reads (`mode:"data"`, `actions:["read"]`) are intentionally skipped by the IAM extractor. (Round-6 P5.) | Low | **Bounded:** a data source cannot itself create/modify cloud resources via the terraform graph; the risk is the *program's* side effects, which are the same unauditable-command class as `local-exec` (now gated as `UNAUDITABLE_PROVISIONER`). An advisory when `external` data sources are present is a candidate follow-up. | When the provisioner/`external` advisory is broadened. |
| R-017 | **AWS/Azure effective-access does not yet bind the cred identity to an EXPECTED account.** GCP fails closed on a cred-project ≠ scope-project mismatch (round-6 E3; round-7 R7-2 also derives the project from `client_email` when the SA key omits `project_id`, so the binding can't silently disable itself). AWS now has a *real* effective-access path (`aws iam simulate-principal-policy`, round-7 R7-1) and Azure lists role assignments, but neither yet asserts the resolved cred belongs to the intended account/subscription. | Low | **Bounded:** requires a *misconfigured or substituted* cred (already a strong position), and the verifier never claims an account identity it didn't check — it reports what the cred sees (`az`/`aws` returning `[]`/`{}` for a wrong-but-empty account is the residual surface). Closing needs the caller to pass an expected account/subscription so the verifier can assert `sts:GetCallerIdentity` (AWS) / `--subscription` (Azure) matches and fail closed otherwise. | When `verify-access` accepts an expected-account parameter from the caller/CI. |

## Closed

| ID | Finding | Severity | Resolution |
|----|---------|----------|------------|
| R-004 | Operator liveness endpoint (`:8080/healthz`) reachable by trusted co-located pods (no ingress NetworkPolicy fronted it). | Info | **Fixed (v1.0.0 track, feat/k8s).** `deploy/operator/operator.yaml` now ships a default-deny-ingress NetworkPolicy on the operator (`podSelector` = the operator, `policyTypes:[Ingress]`, `ingress: []`), so `:8080` is reachable only by the kubelet (host-sourced probes bypass NetworkPolicy) — no co-pod can reach it. Confirmed by R7 red-team. |
| R-001 | Hosted `/all` returned the whole brain to any bearer-authenticated peer (cross-backend HTTP layer in `src/verel/memory/hosted.py`). | Low | **Fixed in v0.51.1.** In signed-writes mode an **unscoped** `/all` (full-brain dump of every scope/principal) requires the **cluster credential** (`X-Cluster-Token`); a **scoped** `/all` stays a normal bearer read (the scope-lattice recall + consolidation use it, and a client could already `/recall` that scope). Legacy single-trust mode unchanged. Pinned by `test_signed_mode_unscoped_all_requires_cluster_credential`; a focused adversarial review ran live coercion exploits (empty/null/wildcard/list scope) and confirmed the boundary holds (all backends use exact-match scope equality; gate and sink read the same value → no parse differential). |
